# ledgr MVP — Result Spec

## Outcome

`ledgr` is a Python CLI. Given a plain-text ledger file of buy and sell transactions for
CAD-listed securities (ETFs, bonds, stocks), running it prints a table to stdout showing,
per holding: quantity currently held, remaining cost basis, current market price, current
market value, and annualized money-weighted rate of return (XIRR) across the holding's
full transaction history. A final TOTAL row shows aggregate cost basis, aggregate current
value, and the portfolio-wide XIRR across every transaction in the ledger. Current prices
come from a live lookup against Yahoo Finance's public chart endpoint at run time (direct
HTTP via `requests`, not the `yfinance` package — see Revision note below). The Yield
figure reflects price appreciation only — it does not include dividends or coupon
payments, and the table makes this explicit.

**Revision note (post-planning, 2026-06-23):** the original spec named the `yfinance`
package as the price source. Verified live during planning that `yfinance` v1.4.1
unconditionally pulls in `numpy` and `pandas` as transitive dependencies (via `uv tree`),
which makes Closeness check 11 ("no numpy/scipy anywhere in the resolved dependency tree")
unsatisfiable for as long as `yfinance` is a dependency, regardless of which XIRR
algorithm is chosen. Confirmed with the user: replace `yfinance` with direct calls to
Yahoo Finance's public chart JSON endpoint
(`https://query2.finance.yahoo.com/v8/finance/chart/{ticker}`) using only `requests`.
Verified live: returns `currency`, `regularMarketPrice`, and historical `close` prices for
CAD tickers (`XEQT.TO`, `VAB.TO`) and USD tickers (`VOO`); returns a clean HTTP 404 with a
JSON error body for an unresolvable ticker. This keeps the original "free, keyless, live
Yahoo Finance lookup" intent while adding zero numpy/pandas footprint. All closeness
checks below that reference "yfinance" mean "the Yahoo chart endpoint integration"
throughout — wording left as-is elsewhere since behavior is unchanged, only the HTTP
client is swapped.

## Ledger format

One transaction per line, either:

```
buy <TICKER> <QUANTITY> @ <PRICE> on <YYYY-MM-DD>
sell <TICKER> <QUANTITY> @ <PRICE> on <YYYY-MM-DD>
```

- `TICKER` — a yfinance-resolvable symbol for a CAD-listed security (e.g. `XEQT.TO`).
- `QUANTITY` — positive number, fractional allowed (e.g. `12.5`).
- `PRICE` — positive decimal, CAD: price paid (buy) or price received (sell) per unit.
- `YYYY-MM-DD` — ISO date, must not be in the future relative to the run time.
- Blank lines and lines starting with `#` are ignored.
- Transactions are processed in chronological date order regardless of the order lines
  appear in the file.
- Any line that doesn't match this grammar is a parse error (see Closeness check 6).
- A `sell` for a quantity greater than what's held as of that date, for that ticker, is a
  fatal error (see Closeness check 13).

Cost basis on a partial sell is tracked using **average cost basis (ACB)** — a single
pooled cost-per-unit for each ticker, reduced proportionally on every sell. This matches
the Canadian adjusted-cost-base convention and avoids per-lot (FIFO) tracking. There is no
separate realized-gain/loss column; the realized portion of any sale is reflected only
through the XIRR cash-flow series (a sell is a positive cash flow at its date).

Example:

```
# my taxable account
buy XEQT.TO 10 @ 28.40 on 2023-03-01
buy XEQT.TO 5 @ 31.10 on 2024-01-15
sell XEQT.TO 6 @ 33.00 on 2024-09-01
buy VAB.TO 20 @ 25.75 on 2023-06-10
```

## Invocation

```
ledgr <path-to-ledger-file>
```

No subcommands, no default ledger path/discovery, no config file for v1.

## Non-goals

- Per-lot (FIFO) cost-basis tracking, or a user-configurable choice of cost-basis method
  — average cost basis is the only method in v1.
- A separate realized-gain/loss column — realized amounts are reflected only via the
  Yield (XIRR) calculation.
- Dividend/coupon tracking — yield reflects price appreciation only (and the table says
  so explicitly, per Closeness check 12).
- Multi-currency / FX conversion — every ticker is assumed CAD-listed; no currency field
  in the ledger grammar, but the tool verifies the resolved listing currency itself (see
  Closeness check 14).
- JSON or any machine-readable output mode.
- Transaction fees/commissions as a separate field (fold into price paid/received, if at
  all).
- Persisting fetched prices or caching across runs.
- Packaging/publishing to PyPI as part of this result (repo + working CLI is the result).
- Auto-discovering a ledger file; the path is always an explicit argument.

## Closeness checks

1. Running `ledgr <file>` against a ledger with two holdings (each with multiple buys at
   different dates and prices) prints one row per distinct ticker plus a final TOTAL row,
   to stdout, with columns: Ticker, Quantity, Cost Basis, Current Price, Current Value,
   Yield (XIRR %).
2. Per-ticker Yield is computed as the annualized XIRR over that ticker's full cash-flow
   history: one negative cash flow per buy (`qty * price`, dated at the buy date), one
   positive cash flow per sell (`qty * price`, dated at the sell date), and — if any
   quantity remains held — a final positive cash flow dated today sized
   `qty_held * current_price`.
3. The TOTAL row's Yield is the annualized XIRR over the combined cash-flow series of
   every transaction across every ticker in the ledger, plus one positive cash flow dated
   today sized at total current portfolio value (sum of all tickers still holding
   quantity).
4. Current price for each ticker that still has quantity held is fetched live via
   yfinance at run time — not read from the ledger, not cached from a prior run.
5. A ledger with a single buy line and no sells, when run, produces a Yield that matches
   a hand-computed XIRR for that single cash flow pair within 0.01 percentage points
   (verifiable: for a single buy with no sells, XIRR reduces to
   `(current_value/cost_basis)^(1/years) - 1`).
6. A ledger file containing malformed lines (e.g. missing `@`, bad date format, negative
   quantity, future date, unrecognized verb) causes the tool to exit non-zero and print
   an error identifying every malformed line's number and content in a single error
   report — not just the first one found. No table is printed.
7. A ledger file containing a ticker yfinance cannot resolve causes the tool to exit
   non-zero with an error naming the unresolvable ticker. No table is printed, even if
   other tickers in the same ledger resolve fine.
8. An empty ledger file (no transaction lines) causes the tool to exit non-zero with an
   error stating no transactions were found.
9. `ledgr --help` prints usage including the invocation form and an example line.
10. `uv run ledgr <file>` works from a clean checkout after `uv sync`, with no manual
    venv/dependency steps beyond that.
11. Inspecting the fully resolved dependency tree (`uv.lock`, or `uv tree` output) — not
    merely the direct `dependencies` array in `pyproject.toml` — shows no `scipy`,
    `numpy`, or other heavy numerical dependency anywhere, direct or transitive. XIRR is
    computed with a lightweight root-finder (e.g. hand-rolled Newton's method or
    bisection) or a small dedicated package with no such transitive dependency.
12. The table output includes a visible note (a column sub-header, footer line, or
    equivalent) stating that Yield reflects price appreciation only and excludes
    dividends/coupon payments.
13. A ledger where a `sell` line's quantity exceeds the quantity held for that ticker as
    of that date causes the tool to exit non-zero with an error naming the ticker, the
    line, and the oversell amount. No table is printed.
14. A ledger containing a ticker whose yfinance-resolved listing currency is not CAD
    causes the tool to exit non-zero with an error naming the ticker and its resolved
    currency. No table is printed, even if other tickers resolve as CAD.
15. A ledger where a ticker has been fully sold (zero quantity remains held as of today)
    still produces a row for that ticker, with Quantity and Current Value shown as zero
    and Yield reflecting the fully realized XIRR across its complete buy/sell history —
    without any live price fetch attempted for that ticker.
16. A ledger with at least 3 buys at irregular, unevenly-spaced dates including a
    simulated deep-drawdown-then-partial-recovery price pattern produces an XIRR matching
    an independently computed reference value (e.g. cross-checked against a spreadsheet
    XIRR function or a reference implementation) within 0.01 percentage points. If the
    root-finder fails to converge, it reports that failure explicitly rather than
    returning an unguarded iteration result.

## Constraints

- Single ledger file per run; no merging multiple files in v1.
- Output is plain-text table only; no `--json` flag in v1 (may be added later, not part
  of this result).
- "Today" for the final cash flow is the system clock date at run time.
- A price-fetch failure is retried at least once (bounded, short backoff) before being
  classified as an unresolvable ticker; the fatal error text distinguishes "no data
  returned after retry" from a malformed/unparseable response.
- No silent skip/fallback for any fatal condition (malformed line, oversell, unresolvable
  ticker, non-CAD currency, empty ledger) — all are fatal errors that stop the run.

## Assumptions

- The machine running `ledgr` has live internet access to reach Yahoo Finance's public
  chart endpoint.
- **Verified live during planning (2026-06-23):** Yahoo's chart endpoint
  (`/v8/finance/chart/{ticker}`) is free, keyless, and resolves `.TO`-suffixed CAD-listed
  tickers (`XEQT.TO`, `VAB.TO`) as well as USD tickers (`VOO`). Each response's
  `chart.result[0].meta` object carries both `currency` and `regularMarketPrice` in one
  call — no second request needed for the currency guard (Closeness check 14). An
  unresolvable ticker returns HTTP 404 with a `chart.error` JSON body, not a generic
  network error — distinguishable from a transient failure (timeout, 5xx, malformed JSON).
- "Average yearly rate of growth" means money-weighted return (XIRR), not time-weighted
  return (TWR) — confirmed earlier in conversation as the correct metric for cash flows
  spread across multiple transaction dates.
- Fractional share quantities are valid and common enough to support from day one.
- Average cost basis (ACB) is the right cost-basis method for this user's CAD-listed,
  Canadian-context portfolio — matches Canadian tax convention, not an arbitrary pick.

## Parking lot

- Per-lot (FIFO) cost-basis tracking as a user-selectable alternative to ACB.
- Dividend/coupon ledger entries folded into the XIRR cash-flow series.
- Multi-currency ledger entries + FX rate lookup to a reporting currency.
- `--json` output flag for scripting.
- Caching/memoizing price lookups within or across runs.
- Pluggable/alternate price source if yfinance proves unreliable.
- Publishing the package to PyPI under the `ledgr` name.
- Specific XIRR algorithm choice (Newton's method vs. bisection vs. a small library) —
  left to the planning stage, constrained only by "no scipy/numpy in the resolved
  dependency tree."
- Specific retry count/backoff duration for transient price-fetch failures — left to
  planning; only the existence of a retry is spec-level.
- A separate "lint the ledger" command independent of running a full report.
